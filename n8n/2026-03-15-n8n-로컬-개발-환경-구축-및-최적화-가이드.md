### Context
* n8n은 대규모 모노레포 구조를 가진 프로젝트로, 로컬 환경에서 소스 코드를 직접 실행하고 수정하기 위해서는 정교한 개발 환경 설정이 필요합니다.
* 특히 네이티브 바인딩(Native Binding) 모듈인 `sqlite3`와 `isolated-vm`은 시스템 아키텍처와 Node.js 버전에 강하게 종속되어 있어 별도의 재빌드(Rebuild) 과정이 필수적입니다.
* 이번 문서는 `fnm`을 활용한 노드 버전 자동 관리, `pnpm` 기반의 의존성 설치, 그리고 로컬 실행 시 발생할 수 있는 보안 및 권한 문제를 해결하기 위한 환경 변수 최적화 방법을 다룹니다.

### Core
* 환경 세팅 및 Node.js 버전 관리
  * `brew install fnm` (Node.js 버전 관리자 설치)
  * `echo 'eval "$(fnm env --use-on-cd)"' >> ~/.zshrc` (디렉토리 진입 시 자동 전환 설정)
  * `source ~/.zshrc`
  * `fnm install 22` (n8n 개발을 위한 Node 22 설치)
  * `npm install -g pnpm@10.22.0 node-gyp` (패키지 매니저 및 네이티브 빌드 도구 설치)

* 초기 빌드 및 네이티브 모듈 수동 컴파일
  * `echo "22" > .node-version` (자동 버전 전환 파일 생성)
  * `pnpm install` (의존성 설치)
  * `pushd node_modules/.pnpm/sqlite3@5.1.7/node_modules/sqlite3 && node-gyp rebuild && popd` (SQLite3 재빌드)
  * `pushd node_modules/.pnpm/isolated-vm@6.0.2/node_modules/isolated-vm && node-gyp rebuild && popd` (Isolated-VM 재빌드)
  * `pnpm build` (전체 프로젝트 빌드)

* n8n 로컬 개발용 실행 설정
  * `N8N_SECURE_COOKIE=false`: 로컬 HTTP 환경에서 쿠키/세션 전송 허용
  * `N8N_DEFAULT_LOCALE=ko`: 인터페이스 한국어 설정
  * `N8N_DIAGNOSTICS_ENABLED=false`: 진단 데이터 전송 비활성화
  * `N8N_TEMPLATES_ENABLED=false`: 공식 템플릿 통신 차단으로 부팅 최적화
  * `N8N_RESTRICT_FILE_ACCESS=false`: 로컬 파일 시스템 접근 제한 해제

### Insight
* `fnm`의 `--use-on-cd` 옵션은 `.node-version` 파일을 감지하여 런타임을 자동 전환하므로, 여러 프로젝트를 병행할 때 발생할 수 있는 노드 버전 불일치 문제를 근본적으로 방지합니다.
* n8n의 핵심 라이브러리인 `isolated-vm`과 `sqlite3`는 사전에 컴파일된 바이너리가 환경과 맞지 않을 때 오류를 발생시키므로, `node-gyp rebuild`를 통해 로컬 아키텍처(M 시리즈 맥 등)에 맞게 수동 재빌드하는 과정이 개발 안정성을 좌우합니다.
* 로컬 환경에서 `N8N_SECURE_COOKIE`를 `true`(기본값)로 둘 경우 브라우저 보안 정책에 의해 로그인이 차단될 수 있으며, 모든 경로의 파일 제어가 필요한 자동화 워크플로우를 개발할 때는 `N8N_RESTRICT_FILE_ACCESS` 설정을 통해 개발 편의성을 높일 수 있습니다.
