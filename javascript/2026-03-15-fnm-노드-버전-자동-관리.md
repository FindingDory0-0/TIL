### Context
개발 환경에서 프로젝트마다 요구하는 Node.js 버전이 다를 경우, 버전을 수동으로 전환하는 것은 번거롭고 실수하기 쉽습니다. fnm(Fast Node Manager)은 Rust로 작성되어 매우 빠른 속도를 자랑하며, .node-version 파일을 사용하여 프로젝트 폴더 진입 시 자동으로 Node.js 버전을 전환하는 기능을 제공합니다. 이 과정에서 발생할 수 있는 환경 설정 이슈와 PATH 우선순위 문제를 해결하는 방법을 정리합니다.

### Core
fnm을 활용한 자동 버전 전환을 설정하고, 기존 패키지 관리자와의 충돌을 방지하는 설정은 다음과 같습니다.

* shell 설정 (.zshrc 또는 .bashrc):
```bash
# fnm 환경 변수 및 자동 전환 설정
eval "$(fnm env --use-on-cd)"
```

* 프로젝트 설정:
프로젝트 루트 디렉토리에 `.node-version` 파일을 생성하고 원하는 버전 번호를 입력합니다.
```text
# .node-version 파일 예시
20.18.0
```

* 문제 해결 (PATH 우선순위):
Homebrew 등을 통해 설치된 Node.js가 PATH 상에서 fnm보다 우선순위가 높을 경우, fnm의 제어권이 무시될 수 있습니다. 이 경우 .node-version 파일을 통해 fnm이 명시적으로 버전을 제어하도록 강제하거나, shell 설정 파일에서 fnm의 PATH를 최상단으로 올리는 것이 좋습니다.

### Insight
fnm은 기존 nvm에 비해 속도 면에서 큰 이점을 제공하며, Rust 기반이라 안정적입니다. 특히 --use-on-cd 옵션은 개발자의 생산성을 크게 높여줍니다. 다만, 여러 Node 관리 도구가 시스템에 공존할 경우 PATH 오염으로 인한 충돌이 잦으므로, 시스템 전체에 설치된 Node(Homebrew 등)를 삭제하고 fnm으로 일원화하는 것을 권장합니다.
