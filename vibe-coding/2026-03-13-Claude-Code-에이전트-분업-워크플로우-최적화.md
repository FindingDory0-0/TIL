### Context
Claude Code를 사용할 때 개발 생산성을 높이기 위해 tmux를 사용하여 여러 세션을 분할하고 각 세션에서 독립적으로 Claude 에이전트를 실행하는 방식을 시도했습니다. 하지만 각 세션이 격리(isolation)되어 있어 에이전트 간의 컨텍스트 공유가 불가능하고, 작업 결과를 수동으로 파일에 기록하여 동기화해야 하는 등 오히려 워크플로우가 파편화되는 문제를 경험했습니다.

### Core
단일 세션 내에서 Claude Code의 에이전트 생성 기능을 활용하여 작업을 분업하는 것이 훨씬 효율적입니다. 다음은 단일 콘솔 환경에서 에이전트에게 역할을 부여하여 작업을 위임하는 예시 접근 방식입니다.

```bash
# Claude Code가 실행 중인 단일 세션에서 명시적으로 에이전트 분업 지시
"너는 현재 프로젝트의 설계자 역할을 맡아줘. 
각 모듈별로 상세 구현을 담당할 하위 에이전트들을 생성하고, 
그들에게 작업을 할당한 뒤 최종 결과물을 나에게 취합해서 보고해줘."
```

단일 세션 내에서는 다음과 같은 이점이 있습니다.
* 공유된 메모리: 동일한 세션 내에서 에이전트 간의 문맥(context)이 유지됩니다.
* 실시간 콜백: 작업 결과를 별도의 파일로 저장할 필요 없이 에이전트 간 즉각적인 리턴이 가능합니다.
* 통합된 관리: 전체 작업 흐름을 하나의 스레드에서 추적하고 관리할 수 있습니다.

### Insight
초기에는 tmux를 이용한 병렬 처리가 효율적이라 생각했으나, Claude Code와 같은 LLM 기반 에이전트는 '문맥 유지'가 핵심입니다. 독립된 세션은 에이전트의 지능을 파편화하지만, 하나의 세션 내에서 에이전트에게 명확한 '역할(Role)'을 부여하여 분업을 지시하는 방식은 에이전트의 추론 능력을 극대화합니다. 향후 복잡한 기능 구현 시에는 쉘 분할보다는 에이전트 체이닝(Agent Chaining) 방식을 통해 워크플로우를 최적화하는 것이 바람직합니다.
